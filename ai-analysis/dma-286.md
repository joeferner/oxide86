# DMA 286 — Second Controller (Channels 4–7)

## Problem

CheckIt DMA diagnostics pass on an 8086 machine but fail on 286. A 286 PC/AT has
two cascaded 8237A DMA controllers:

- **DMA1** (channels 0–3): 8-bit transfers, ports 0x0000–0x000F — already working.
- **DMA2** (channels 4–7): 16-bit transfers, ports 0x00C0–0x00DF — ports are
  accepted by the emulator but the controller is never initialised by the BIOS and
  CASCADE mode is not handled in `tick()`, so DMA2 channels never advance.

## Root Cause Analysis

### 1. `bios_dma_reset` only initialises DMA1

`core/src/cpu/bios/mod.rs` — `bios_dma_reset` sets up channel 0 (memory refresh)
but does nothing for DMA2. A real 286 AT BIOS POST also runs:

```
out 0xDA, al  ; DMA2 master clear   (offset 0x0D → port 0xDA)
out 0xD8, al  ; DMA2 clear flip-flop (offset 0x0C → port 0xD8)
mov al, 0xC0  ; cascade mode, ch 0 of DMA2
out 0xD6, al  ; DMA2 mode register  (offset 0x0B → port 0xD6)
mov al, 0x00  ; unmask, ch 0
out 0xD4, al  ; DMA2 single mask    (offset 0x0A → port 0xD4)
```

Without this, DMA2's mask register stays 0x0F (all channels masked) and nothing
can run.

### 2. CASCADE mode not handled in `Dma8237::tick()`

`core/src/devices/dma.rs` — When DMA2 channel 4 (DMA2 internal channel 0) is
programmed with mode `0xC0` (bits 7-6 = 11 = CASCADE), `tick()` has no guard for
this. It would generate spurious `DmaTransfer` entries on every tick because
`dreq` bit 0 of DMA2 is permanently set (simulating DMA1's HRQ cascading in).

### 3. `Bus` / `bios_reset` do not know the CPU type

`bios_reset(bus: &mut Bus)` is CPU-agnostic. `BusConfig` and `Bus` have no
`cpu_type` field, so `bios_dma_reset` cannot conditionally initialise DMA2 for
286-class machines.

### 4. No test for DMA2 counter advancement

All existing DMA assembly tests target DMA1 and use `[CPU 8086]`. There is no
test that verifies DMA2 channels advance their counters.

## DMA2 Port Map (quick reference)

DMA2 ports are word-spaced: physical port = 0xC0 + (internal_offset × 2).

| Internal offset | Physical port | Register |
|---|---|---|
| 0x00 | 0xC0 | ch4 (ch0) address |
| 0x01 | 0xC2 | ch4 (ch0) count |
| 0x02 | 0xC4 | ch5 (ch1) address |
| 0x03 | 0xC6 | ch5 (ch1) count |
| 0x04 | 0xC8 | ch6 (ch2) address |
| 0x05 | 0xCA | ch6 (ch2) count |
| 0x06 | 0xCC | ch7 (ch3) address |
| 0x07 | 0xCE | ch7 (ch3) count |
| 0x08 | 0xD0 | command (W) / status (R) |
| 0x09 | 0xD2 | request register (software DREQ) |
| 0x0A | 0xD4 | single channel mask |
| 0x0B | 0xD6 | mode register |
| 0x0C | 0xD8 | clear flip-flop |
| 0x0D | 0xDA | master clear (W) / temp reg (R) |
| 0x0E | 0xDC | clear mask register (unmask all) |
| 0x0F | 0xDE | write all masks |

## Implementation Plan (Test-First)

### ✅ Step 1 — Write the failing test assembly

Create `core/src/test_data/devices/dma/counter_advances_286.asm`:

```nasm
; DMA2 channel 5 counter-advance test (286 system)
;
; Programs DMA2 channel 5 (DMA2 internal ch 1) in single/verify mode,
; software-asserts DREQ via the request register, then polls its
; current-count register until it changes from the initial value.
;
; DMA2 channel 4 is the cascade channel (connects DMA1 HRQ).
; Channels 5-7 are free for this test.
;
; Port map used:
;   0xDA  DMA2 master clear
;   0xD8  DMA2 clear flip-flop
;   0xD6  DMA2 mode register
;   0xD4  DMA2 single channel mask
;   0xD2  DMA2 request register (software DREQ)
;   0xC4  ch5 base/current address (DMA2 ch1 addr)
;   0xC6  ch5 base/current count  (DMA2 ch1 count)
;
; Mode byte 0x41:
;   bits 7-6: 01 = single mode
;   bit  5:   0  = address increment
;   bit  4:   0  = no auto-init
;   bits 3-2: 00 = verify (no data movement)
;   bits 1-0: 01 = channel 1 (DMA2 internal → global ch 5)

[CPU 286]
org 0x0100

start:
    ; DMA2 master clear
    mov al, 0x00
    out 0xDA, al

    ; Clear flip-flop
    out 0xD8, al

    ; ch5 base/current address = 0x0000
    out 0xC4, al        ; low byte
    out 0xC4, al        ; high byte

    ; ch5 base/current count = 0x00FF
    out 0xD8, al        ; clear flip-flop
    mov al, 0xFF
    out 0xC6, al        ; low byte = 0xFF
    mov al, 0x00
    out 0xC6, al        ; high byte = 0x00

    ; Mode: single, increment, no-auto-init, verify, DMA2-ch1
    mov al, 0x41
    out 0xD6, al

    ; Unmask ch5 (bit 2 = 0 = unmask, bits 1-0 = 01 = DMA2 ch1)
    mov al, 0x01
    out 0xD4, al

    ; Software-assert DREQ on ch5 (bit 2 = 1 = set, bits 1-0 = 01)
    mov al, 0x05
    out 0xD2, al

    ; Read initial count low byte
    mov al, 0x00
    out 0xD8, al        ; clear flip-flop
    in  al, 0xC6        ; ch5 count low byte — should be 0xFF
    mov bl, al

    ; Poll until count changes (with timeout)
    mov cx, 0xFFFF

poll_loop:
    mov al, 0x00
    out 0xD8, al        ; clear flip-flop
    in  al, 0xC6        ; read current count low byte
    cmp al, bl
    jne success

    loop poll_loop

    ; Timeout
    mov ah, 0x4C
    mov al, 0x01
    int 0x21

success:
    mov ah, 0x4C
    mov al, 0x00
    int 0x21
```

### Step 3 — Pass `cpu_type` into `bios_dma_reset`

In `core/src/cpu/bios/mod.rs`:

- Change `bios_dma_reset(bus: &mut Bus)` to
  `bios_dma_reset(bus: &mut Bus, cpu_type: CpuType)`.
- Update the call in `bios_reset` to pass `cpu_type`.

`bios_reset` itself must also accept `cpu_type`:

```rust
pub(crate) fn bios_reset(bus: &mut Bus, cpu_type: CpuType) {
    bios_interrupt_handlers_reset(bus);
    bda_reset(bus);
    bios_int15_system_config_reset(bus, cpu_type);
    bios_font_reset(bus);
    bios_dma_reset(bus, cpu_type);
    if bus.video_card().card_type() == VideoCardType::VGA {
        bios_vga_static_func_table_reset(bus);
    }
}
```

Both callers of `bios_reset` in `core/src/bus.rs` (`reset` and `warm_reset`) need
updating. `Bus` must expose or store `cpu_type` so those call sites can supply it.

The simplest approach:
1. Add `cpu_type: CpuType` to `BusConfig`.
2. Store it as a private field on `Bus`.
3. In `Bus::reset` / `Bus::warm_reset` pass `self.cpu_type` to `bios_reset`.
4. Update `Computer::new` to include `cpu_type` in `BusConfig`.

### Step 4 — Initialise DMA2 in `bios_dma_reset` for 286+

```rust
fn bios_dma_reset(bus: &mut Bus, cpu_type: CpuType) {
    // --- DMA1 (unchanged) ---
    bus.io_write_u8(0x000D, 0x00); // master clear
    bus.io_write_u8(0x000C, 0x00); // clear flip-flop
    bus.io_write_u8(0x0000, 0x00); // ch0 addr low
    bus.io_write_u8(0x0000, 0x00); // ch0 addr high
    bus.io_write_u8(0x0001, 0xFF); // ch0 count low
    bus.io_write_u8(0x0001, 0xFF); // ch0 count high
    bus.io_write_u8(0x000B, 0x58); // mode: single|incr|auto-init|verify|ch0
    bus.io_write_u8(0x000A, 0x00); // unmask ch0

    // --- DMA2 (286+ AT only) ---
    if cpu_type.is_286_or_later() {
        bus.io_write_u8(0x00DA, 0x00); // DMA2 master clear
        bus.io_write_u8(0x00D8, 0x00); // DMA2 clear flip-flop
        // Ch4 = cascade: mode 0xC0 (bits 7-6=11, bits 1-0=00 = DMA2 ch0)
        bus.io_write_u8(0x00D6, 0xC0); // DMA2 mode: cascade, ch0
        bus.io_write_u8(0x00D4, 0x00); // unmask DMA2 ch0 (global ch4)
    }
}
```

### Step 5 — Handle CASCADE mode in `Dma8237::tick()`

In `core/src/devices/dma.rs`, inside the `for ch in 0..4` loop of `Dma8237::tick()`:

```rust
// CASCADE mode (bits 7-6 = 11): this channel is a slave pass-through.
// No data transfers are generated; the master controller handles everything.
let mode = self.channels[ch].mode;
if mode & 0xC0 == 0xC0 {
    continue;
}
```

Add this check immediately after the existing mask/dreq guards.
